### 任务1: CLS VIP信息结构化提取

**问题**: VIP文章中的研报信息包含丰富的股票催化概念，但目前仅作为文章列表存储标题和摘要。以"风口研报·公司"类文章为例，其中包含特定股票的深度分析（如"相关股票：科创板1只"），需要：
1. 找出对应的具体股票（通过研报描述匹配）
2. 总结其主要催化概念
3. 提取结构化信息加入VIP信息表
4. VIP信息表作为研报综合推理挖掘的重要数据维度
5. 与研报、钱三强选股结果一起作为交付物推送飞书

**示例输入**:
```
2026-06-25 星期四
10:11【风口研报·公司】算力芯片、光模块、先进封装拉动粉体材料需求...
相关股票：科创板1只
```

**期望输出**: 
- 股票名称和代码（如：有研粉材 688456）
- 核心催化概念（分主线，每条含具体逻辑）
- 附加加分逻辑

**实现思路**:
- 新建 `vip_extractor.py` 模块
- 从 `fetch_data.py` 的 `fetch_cls_pages_via_api()` 返回的VIP文章中筛选"风口研报"类文章
- 对"相关股票：XX只"的描述，用Tushare `stock_basic` API按板块+行业+名称关键词匹配
- 催化概念总结需要AI介入——考虑在 `extract_summary.py` 中生成VIP信息表结构，概念总结在报告撰写阶段由AI完成
- `push_feishu.py` 增加VIP信息表MD文件作为第三个交付物

**涉及文件**:
- 新建: `vip_extractor.py`
- 修改: `extract_summary.py` (新增chapter_vip)
- 修改: `push_feishu.py` (增加VIP信息表推送)
- 修改: `analysis_prompt.md` (增加VIP信息分析指引)
- 修改: `fetch_data.py` (VIP采集后调用提取器)

### 任务2: CLS电报时间段衔接 + 周信息存储

**问题**: 目前电报采集固定取最近50条，仅覆盖过去约1小时。晨报(6:00)、午报(11:30)、晚报(20:00)三个时间点采集的电报在时间上不衔接，存在大量信息盲区。需要实现时间段的延续覆盖。

**核心需求**:
- 三次采集的电报在时间上要衔接，覆盖全天
- 考虑将一周信息结构化标注压缩后暂存仓库，供分析时使用

**实现思路**:
- `fetch_data.py` 的电报采集改为基于时间戳过滤（记录上次采集的最后一电报时间，本次从该时间点开始采集）
- 新建 `data/cls_telegraph_archive/` 目录，按日期存储电报快照
- 每次 `fetch_data.py` 执行时：读取归档中最新时间戳 → 从该时间戳开始采集 → 新电报追加到当日归档文件
- 归档格式: `data/cls_telegraph_archive/YYYY-MM-DD.json`，含完整电报列表+去重
- `extract_summary.py` 读取归档时做时间范围标注（如"本时段新增XX条，全天累计XX条"）
- 周末总结任务可读取整周归档

**涉及文件**:
- 修改: `fetch_data.py` (电报采集改为增量+归档)
- 修改: `extract_summary.py` (读取归档+时间范围标注)
- 修改: `.gitignore` (需决定是否将归档纳入git——建议纳入，因为自动任务每次clone是全新环境)

**注意**: 自动任务每次执行是全新clone环境，本地文件不持久化。因此归档必须推送到GitHub才能跨任务共享。电报数据量需控制（每日约200-400条，每条约200字，单日约80KB，一周约560KB，可接受）。

**实现思路**:

- `push_feishu.py` 推送后自动 `git add reports/ data/data_summary.json data/qian_sanqiang_results.json && git commit && git push`
- 考虑目录结构: `reports/YYYY/MM/YYYY-MM-DD_晨报.md` 按月分目录
- `data_summary.json` 也纳入版本控制（单文件约500行，可接受）
- `raw_data_*.json` 仍然排除（24万行+，太大）

**涉及文件**:
- 修改: `.gitignore`
- 修改: `push_feishu.py` (增加自动commit+push)
- 修改: `SKILL.md` (步骤5增加git push报告)

**风险**: 每次执行增加git操作耗时约5-10秒。报告MD文件单次约10-20KB，一年约5MB，可接受。

### 任务4: 周末自动化任务

**前置依赖**: 任务3（报告和数据已存入仓库，周末任务才能读取整周数据）

**实现思路**:

**涉及文件**:

### 任务5: 分析推理链优化

**问题**: 目前版本的推理、联想和洞见不足。提示词随迭代越来越庞大，导致:
- 研究能力和思维弱化（模型被过多规则约束）
- 过拟合（过度适配当前提示词模板）
- 创新性不足（金股推荐倾向于重复已知标的）

**核心矛盾**: 提示词需要足够详细来约束输出格式和数据准确性，但过度的约束会压制模型的推理能力。

**实现思路**:
- 分层提示词架构: 将 `analysis_prompt.md` 拆分为
  - `prompt/rules.md` — 硬约束（格式、数据来源、红线）——不可违反
  - `prompt/reasoning.md` — 软引导（推理方法、分析框架、联想方向）——参考性
  - `prompt/examples.md` — 范例库（优秀报告片段）——学习性
- 减少硬约束中的"必须分析XX"类指令，改为开放式问题引导
- 引入"思维链"引导: 在金股推理部分，要求模型先列出信号→验证→结论的完整链路
- 范例库: 从历史优秀报告中提取3-5个金股推理范例，供模型参考

**涉及文件**:
- 重构: `analysis_prompt.md` → 拆分为 `prompt/` 目录下多个文件
- 新建: `prompt/rules.md`, `prompt/reasoning.md`, `prompt/examples.md`
- 修改: 3个自动任务的步骤3（引用新的提示词文件结构）

**注意**: 这是最需要反复迭代的任务，需要对比修改前后的报告质量。

### 任务6: 强制使用GLM-5.2进行推理

**问题**: 自动任务使用的默认模型为TRAE-Auto-model，推理能力可能不足以支撑深度分析。需探索在需要整合信息和推理的步骤强制使用GLM-5.2。

**调研方向**:
- TRAE自动任务是否支持指定模型？（检查Schedule工具的参数）
- 如果不支持任务级模型指定，是否可以在任务prompt中引导模型自行调用更强模型？
- TRAE的MCP工具或Skill是否支持模型切换？

**实现思路**:
- 调研TRAE自动任务的模型指定能力
- 如果支持: 在自动任务配置中指定推理步骤使用GLM-5.2
- 如果不支持: 探索替代方案（如通过API直接调用GLM-5.2生成报告初稿，再由当前模型校验）

**涉及文件**:
- 取决于调研结果，可能修改自动任务配置或新增调用逻辑

### 任务7: Agent优化训练（研究型任务）

**问题**: 报告分析推理的质量高度依赖大模型本身的能力。日本有个"鱼"AI提供了启发——通过对Agent的优化训练，让普通开源AI大模型的输出结果接近顶尖模型。

**这是研究方向，不是确定性工程任务。**

**调研方向**:
- "鱼"AI的具体方法（需WebSearch调研）
- Few-shot learning / In-context learning 在Agent优化中的应用
- 是否可以通过构建训练数据集（历史优秀报告+评分）来优化提示词
- TRAE是否支持自定义Agent训练或微调

**实现思路**:
- Phase 1: 调研"鱼"AI方法和相关技术
- Phase 2: 评估TRAE平台的Agent训练能力
- Phase 3: 设计实验方案（如A/B测试不同提示词策略的报告质量）
- Phase 4: 如果可行，构建训练流程

**注意**: 此任务高度探索性，可能需要多个对话轮次推进。建议在任务1-6完成后再开始。

## 四、技术决策点

以下决策需要在执行过程中确认:

1. **电报归档是否推送到GitHub**: 推荐是。自动任务每次clone全新环境，不推送则跨任务无法共享。数据量可控（每周约560KB）。

2. **VIP股票匹配方式**: Tushare stock_basic API按板块+行业+关键词匹配？还是通过WebSearch搜索研报标题？前者更快但可能不够准确，后者更准但耗时。

3. **提示词拆分粒度**: 拆成3个文件是否足够？还是需要更细的分文件（如按章节拆分）？

4. **周报模式的数据范围**: 整周归档数据量可能较大（5天×3次×24万行raw_data），需要在extract_summary层面做聚合而非传递原始数据。

5. **GLM-5.2的可用性**: 需确认TRAE平台是否支持GLM-5.2，以及自动任务是否可以指定模型。
