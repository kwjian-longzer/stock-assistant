# Agent优化训练研究报告（任务7）

> 调研日期: 2026-06-25  
> 性质: 探索性研究，非确定性工程任务

---

## 一、调研背景

报告分析推理的质量高度依赖大模型本身的能力。计划中提到日本有个"鱼"AI提供了启发——通过对Agent的优化训练，让普通开源AI大模型的输出结果接近顶尖模型。

## 二、"鱼"AI = Sakana Fugu 调研

### 核心概念

"鱼"AI 即 Sakana Fugu（河豚），由日本AI独角兽 Sakana AI 于 2026年6月22日发布 [1]。它不是传统大模型，而是**编排器模型（Orchestration Model）**——不自己回答问题，而是调度其他AI模型（GPT、Claude、Gemini）协作完成任务。

### 四大机制 [1]

1. **识别问题类型**：判断任务是代码/数学/推理/信息检索/科学分析/多模态
2. **选择Worker模型**：根据任务类型选择最擅长的模型
3. **设计Agent工作流**：任务拆分→子任务分配→上下文共享→答案合成
4. **反馈优化**：用真实任务结果反向优化编排策略（强化学习+进化算法）

### 核心洞察

- **不训练更强的基座模型，而是训练"调度能力"**
- 模型能力模块化：新模型发布后可直接加入Worker池
- 用户可控：可根据成本/隐私/合规配置模型池
- **AI竞争从"单一模型能力"扩展到"系统组织能力"** [1]

### 局限性

- 多模型编排带来更高成本和延迟
- 错误归因复杂（路由错误？Worker模型错误？综合过程错误？）
- 编排器本身可能有偏差（错误判断任务类型）

## 三、对Stock-Assistant项目的适用性分析

### 当前架构

```
单一模型(DeepSeek-V4-Pro) → 读取data_summary.json → 撰写报告 → 推送飞书
```

### 潜在改进方向

#### 方案A：模型路由（轻量级，推荐v2.1尝试）

不引入完整编排器，而是在关键步骤做模型路由：

| 步骤 | 当前 | 优化方向 | 理由 |
|------|------|---------|------|
| 数据采集 | Python脚本 | 不变 | 脚本无需AI |
| 摘要提取 | Python脚本 | 不变 | 脚本无需AI |
| 电报信号分析 | DeepSeek | GLM-5.2 | 中文优势+长上下文 |
| VIP研报分析 | DeepSeek | GLM-5.2 | 长文本+中文优势 |
| 金股推理链 | DeepSeek | DeepSeek(保持) | 深度推理优势 |
| 周报撰写 | DeepSeek | GLM-5.2 | 1M上下文加载全周数据 |
| 报告校验 | Python脚本 | 不变 | 脚本无需AI |

**成本影响**：GLM-5.2调用约30次/月（仅周报+VIP分析），月增成本约¥20

#### 方案B：提示词优化（零成本，立即可做）

借鉴Fugu的"反馈优化"思路，但不引入多模型：
1. 收集历史报告+用户反馈（优质/需改进）
2. 构建few-shot examples嵌入analysis_prompt.md
3. 在提示词中加入"优秀报告示例"和"常见错误避免"

**实现路径**：
- v2.0已实现的推理链质量自检清单（任务5）是第一步
- 下一步：从reports/目录中筛选3-5篇高质量报告作为few-shot参考
- 在analysis_prompt.md中增加"优秀报告特征"描述

#### 方案C：完整编排器（远期v3.0+）

部署类似Fugu的编排层，需要：
1. 多模型API接入（DeepSeek + GLM-5.2 + 可能的Claude）
2. 任务路由逻辑（基于任务类型选择模型）
3. 结果整合与校验
4. 成本控制机制

**不推荐短期实施**：复杂度高、成本高、错误归因难

## 四、推荐实施路径

### Phase 1: 提示词优化（v2.0已完成，零成本）
- ✅ 推理链6步规范化（任务5）
- ✅ 质量自检清单（任务5）
- ✅ VIP交叉验证维度（任务1+5）
- ⬜ 下一步：从历史报告中提取few-shot examples

### Phase 2: 模型路由实验（v2.1，低成本）
- 在周报撰写中试用GLM-5.2（利用1M上下文）
- 在VIP研报分析中试用GLM-5.2
- A/B测试：对比DeepSeek和GLM-5.2的报告质量
- 评估指标：用户反馈 + 推理链完整性 + 数据引用准确率

### Phase 3: 编排器探索（v3.0+，高成本）
- 如果Phase 2证明多模型有价值
- 设计轻量级编排逻辑（非完整Fugu复刻）
- 重点：任务路由 + 结果校验 + 成本控制

## 五、TRAE平台能力评估

需进一步确认：
1. TRAE是否支持在自动任务中切换不同模型？
2. TRAE的Schedule任务是否可以指定模型？
3. TRAE是否支持自定义Agent训练/微调？

当前自动任务中已嵌入"【模型要求】执行分析步骤时请使用最强推理模型（DeepSeek-V4-Pro）"指令，说明模型选择有一定灵活性。

## 六、结论

1. **Fugu的"编排器"理念有启发**，但完整实施过于复杂
2. **推荐先做提示词优化（已完成）+ 模型路由实验（v2.1）**
3. **GLM-5.2在长文本和中文场景有优势**，适合周报和VIP分析
4. **DeepSeek在深度推理上有优势**，适合金股推理链
5. **编排器方案留待v3.0+评估**
